### 2026-05-18 レシピ保存・バージョン採用時の重複防止

- **問題**: レシピシステムで保存やバージョン管理操作後に、レシピ内の明細行が勝手に増えたように見えるケースがあった。
- **確認結果**:
  - 通常保存API `/api/recipe/save` は `recipes` 本体に対して `update` のみ実行しており、保存ボタン自体がレシピ本体を新規作成する経路はない。
  - ただし詳細画面の保存ボタンに保存中ロックがなく、未保存の新規明細がある状態で連打すると同じ `newItems` が複数回insertされる余地があった。
  - バージョンプレビューから「採用して保存」する経路では、画面上の明細が `temp-preview-*` になっているため既存DB明細の削除対象が空になり、スナップショット明細を追加insertして重複する可能性があった。
  - `/recipe/new` は `useState` の作成中フラグで二重作成を防いでいたが、Reactの二重実行タイミングでは同期的なガードにならない可能性があった。
- **対応**:
  - レシピ詳細画面に `isSaving` を追加し、保存中は上部/下部の保存・キャンセルボタンを無効化
  - 保存中はボタン表示を「保存中...」に変更して二重送信を防止
  - バージョン採用時はDB上の現在の `recipe_items` を再取得し、既存明細を全削除してからスナップショットをinsertするよう修正
  - `/recipe/new` の作成中ガードを `useRef` に変更し、同一画面内の二重作成を同期的に防止
  - 最終保存方式に合わせ、未確定履歴は1保存につき1件に限定し、UI文言を「履歴メモ」へ変更
  - DB読み取り確認で、直近レシピ本体の同名増殖は見当たらない一方、完全一致の重複明細が19グループ存在することを確認（データ削除は未実施）
- **履歴運用ルール**:
  - 通常編集: 編集 → 必要なら「+ 履歴メモ」 → 最後に青い「保存」で確定
  - 大きな変更前: まず「変更前」などの履歴メモを付けて保存し、そこから編集を始める
  - 途中段階を複数残したい場合: 各段階ごとに「履歴メモ」→「保存」を1セットで実施
  - 過去履歴はクリックでプレビュー、戻す場合だけ「Ver.xを採用して保存」を押す
- **関連ファイル**: `app/recipe/[id]/page.tsx`, `app/recipe/new/page.tsx`
- **確認**: `npm run build` 成功

### 2026-05-18 WEB販売AIデータアナリストの途中月判定強化

- **問題**: WEB販売管理システムのAIチャット（Gemini 2.5 Pro）が、15日取り込み時点の当月途中データを前月・前年同月の月末確定データと単純比較し、「売上半減」「急落」など過剰に危機判定していた。
- **原因**: AIプロンプトに、WEB販売データは原則「15日頃」と「月初」の月2回取り込みであり、当月15日データは途中経過として扱うべき、という業務前提が渡っていなかった。
- **対応**:
  - `/api/web-sales-chat` で対象月が当月の場合、データ状態を「15日取り込み時点の途中データ」または「月初取り込み後の途中データ」と判定
  - AIコンテキストに月内進捗率、評価対象日数、月末着地見込み係数、着地見込み販売数を追加
  - 対象月が途中データの場合、前月・前年同月の月末確定値と単純比較して「半減」「急落」「危機的」と断定しないルールを追加
  - 画面上のAI説明文と初期質問例も、15日取り込み時点は途中経過として扱う文言へ変更
  - 旧AIレポート生成API `/api/web-sales-analyze` 側にも同様の途中データガードを追加
- **関連ファイル**: `app/api/web-sales-chat/route.ts`, `components/web-sales-ai-section.tsx`, `app/api/web-sales-analyze/route.ts`
- **確認**: `npm run build` 成功

### 2026-05-10 広告コンバージョン計測の全面修正 (tsai-sales-db + 各LPリポジトリ)

#### 1. v0用Metaタグ実装プロンプトのロジック修正 (tsai-sales-db)
- **問題**: v0へ「クリック後300ms待ってから遷移」と指示した結果、`e.preventDefault()`と`setTimeout`+`window.open`が生成され、iOS SafariのポップアップブロックやGA4の自動クリック計測阻害という致命的バグを引き起こしていた。
- **対応**: プロンプトの注意書きから待機指示を削除し、以下の厳格な禁止ルールを追加。
  - 「標準のリンク遷移（`e.preventDefault()`等）は絶対に阻害しない」
  - 「`setTimeout`や`window.open`は使用せず、標準の`<a>`タグの`target="_blank"`に任せる」
  - 「イベント送信は`<a>`タグのonClick属性で直接発火させるだけにする」
- **関連ファイル**: `app/web-sales/advertising/lp-tracking-tab.tsx`, `app/lp-tracking/[id]/page.tsx`
- **備考**: 指示文はDBに保存されずテンプレートから動的生成されるため、デプロイにより全品種の生成指示文が一括修正された。

#### 2. 既存LP5品種のコード直接修正・デプロイ
- **問題**: 生成済みの5品種LPに `e.preventDefault()` + `setTimeout` + `window.open` パターンが実装済みで、Google広告のコンバージョン計測が一斉に悪化していた。
- **対応**: 全5リポジトリで以下を実施。
  - `e.preventDefault()` と `setTimeout` / `window.open` を完全削除
  - 標準の `<a href="..." target="_blank">` によるネイティブ遷移に変更
  - GA4 `click_purchase` イベント（`gtag('event', 'click_purchase', {...})`）を全購入ボタンに追加（Yahoo/楽天/Amazon別ラベル付き）
  - Meta Pixel `MallClick` イベントの発火は維持
- **対象リポジトリとLP**:
  | リポジトリ | LP URL | product値 |
  |---|---|---|
  | hajiox/wakeari | chasieu.aizu-syokubura.com | wakeari |
  | hajiox/kitakata | kitakata.aizubrandhall-lp2.com | kitakata |
  | hajiox/akumameshi | akumameshi.aizu-syokubura.com | butameshi |

## 9. psql (PostgreSQLクライアント) のインストール (2026-02-18)

### 9-1. 目的
Supabase REST API (PostgREST) はDDL（ALTER TABLE等のスキーマ変更）を実行できないため、DBマイグレーション時にブラウザでSupabaseダッシュボードのSQL Editorを開く必要があった。これは非効率でAIエージェントとの連携も困難だった。`psql`をローカルにインストールし、コマンドラインから直接SQLを実行可能にする。

### 9-2. インストール内容

| 項目 | 値 |
|------|------|
| ツール | psql (PostgreSQL 17.8) |
| インストール方法 | `winget install PostgreSQL.PostgreSQL.17` |
| インストール先 | `C:\Program Files\PostgreSQL\17\bin\` |
| PATH追加 | ユーザーPATHに追加済み |

### 9-3. 接続コマンド

```powershell
$env:PAGER=""; psql -h aws-0-ap-southeast-1.pooler.supabase.com -p 6543 -U postgres.zrerpexdsaxqztqqrwwv -d postgres --no-psqlrc -P pager=off -c "<SQL>"
```

### 9-4. 注意事項
- `PAGER`環境変数を空にする（Windowsに`cat`コマンドが無いためエラーになる）
- `--no-psqlrc`と`-P pager=off`を必ず付与
- UTF-8エンコーディング確保のため`[Console]::OutputEncoding`を設定
- ワークフロー: `.agent/workflows/db-migration.md`に手順を記載済み

### 9-5. 背景
Supabase REST APIが使えなかった理由:
1. PostgRESTはCRUD操作のみ対応。DDL（ALTER TABLE等）は実行不可
2. `rpc/exec_sql`のようなカスタム関数はデフォルトで存在しない
3. Supabase Management APIは別のアクセストークンが必要

#### 修正したバグ
| # | 問題 | 原因 | 修正内容 |
|---|------|------|----------|
| 1 | 資材・諸経費の使用量入力欄がない | `isMaterialGroup` 条件で使用量欄を非表示（`-`）にしていた | 全タイプで使用量入力欄を表示、プルダウン選択時にデフォルト`usage_amount=1` |
| 2 | 資材の`unit_quantity`が文字列→コスト計算スキップ | materialsテーブルの`unit_quantity`がtext型（"3000"等） | マスター読み込み時に`parseFloat`で数値変換、デフォルト1 |
| 3 | 資材の原価が¥0になる（**根本原因**） | 食材と同じ`usage × (price / unit_quantity)`で計算 → ダンボール¥29.7÷3000=¥0 | **資材・経費**: `usage × price`（1個単価）、**食材**: `usage × (price / unit_quantity)`（g単価）に分岐 |

#### コスト計算式の整理
```
食材:   cost = usage_amount(g) × (price(パック価格) / unit_quantity(パック入数g)) × tax_rate
資材:   cost = usage_amount(個) × price(1個単価) × tax_rate
諸経費: cost = usage_amount(回) × unit_price(1回単価) × tax_rate
```

## 22. WEB/卸販売 単価スナップショット改修・BASE CSVインポート修正 (2026-03-04~05)

### 22-1. 目的
商品価格変更時に過去の売上データが最新価格で再計算される問題を解消し、BASEのCSVインポートでの金額不一致・マッチング問題を修正する。

### 22-2. WEB販売管理 — 単価スナップショット

#### 問題
商品マスターの価格を変更すると、過去月の売上金額も最新価格で再計算されてしまい、過去データの正確性が失われる。

#### 修正
- `web_sales_summary` に `unit_price` カラム追加（インポート時の単価を保存）
- `calculateTotalAllECSites()` のフォールバック: `unit_price → price → productMap.price`
- 過去データは過去の単価で、新規データは現在の単価で正確に計算

### 22-3. 卸販売管理 — 同等修正
- `wholesale_summary` にも `unit_price` カラムを追加
- WEB販売と同じスナップショット方式を適用

### 22-4. OEM売上入力画面 — マスターデータ優先方式
- **問題:** 単価入力フィールドがあるが、結局マスターデータで上書きされて分かりにくい
- **修正:** 単価フィールドをマスターから自動取得・読み取り専用に変更。「単価変更はOEM商品管理から」と注記
- OEMエリアサマリーは `sale.unit_price`（実売上データ）を表示

### 22-5. BASE CSVインポート修正

#### 合計金額の取得（base-parse ver.5）
- CSVの「合計金額」列（23列目）もパースし、商品ごとに金額を集計
- `base_amount` として `web_sales_summary` に保存（参考データ用）

#### BASEのみ複数マッチ許可（csvHelpers.ts）
- **問題:** BASEは「送料別」「送料込み」で同一商品を2つ販売するが、一方がマッチすると他方が宙ぶらりんになる
- **修正:** `channel === 'base'` の場合のみ `matchedIds` 重複排除をスキップ
- 他ECサイトは従来通り1:1マッチを維持

#### 金額表示の設計方針
- クーポン・割引は広告費として扱い、売上金額には影響させない
- 全ECサイト共通: `count × マスター価格 = 本来売り上がった金額` を表示
- `get_monthly_financial_summary` RPC関数も `count × products.price` に統一

```

#### なぜこの方式か

| 方式 | 問題 |
|------|------|
| TSA → DocScanner 直接HTTP | TSAはVercel(HTTPS)、DocScannerはローカル(HTTP)。混合コンテンツでブロックされる。他PCからは `localhost` が自PCを指すため届かない |
| DocScanner → Supabase ポーリング | ✅ 両方がSupabaseにアクセスできるため、ネットワーク制約なし |

CREATE TABLE recipe_print_logs (

  recipe_id uuid NOT NULL,
  recipe_name text NOT NULL,
  category text,
  version_number integer,
  version_note text,
  items jsonb,
  total_cost real,
  selling_price real,
  printed_at timestamptz DEFAULT now(),
  processed boolean DEFAULT false,
  processed_at timestamptz,
  created_at timestamptz DEFAULT now()
);
```

#### 印刷レイアウトのバージョン情報

- プレビュー中: `Ver.3 (配合変更)` のように太字amber色で表示
- 通常時: `最新: v3` と表示

## 36. 裏ラベルチェック: 簡易チェック・画像保存・閾値改修 (2026-03-22)

### 36-2. 簡易チェックモード

**概要:** DB照合をスキップし、OCR結果とユーザ入力の賞味期限で差異判定を行う簡略版。

- **モード選択画面:** 最初に「通常チェック」「簡易チェック」を選択

- **賞味期限入力:** 60日 / 1年 / 18ヶ月 / 2年のプリセット + 自由入力
- **自由入力:** 数値のみ（例: `90`）は「日」として解釈するフォールバック追加
- **判定基準:** 25%差異（2年以上は35%）
- **ダッシュボード表示:** `⚡簡易` バッジで通常チェックと区別

### 36-3. 画像保存機能

**概要:** 通常版・簡易版の両方で撮影写真を自動保存し、ダッシュボードで閲覧可能にする。

#### 画像処理フロー
1. クライアント側: Canvas APIで最大1200pxにリサイズ → JPEG品質段階的低下 → **200KB以下**に圧縮
2. base64でAPIに送信（判定後バックグラウンドで実行、UIをブロックしない）
3. サーバー側: `data/label-images/{checkId}.jpg` に保存
4. DB: `label_checks.image_path` カラムに記録（単一ファイル名 or JSON配列）

#### 複数枚対応
- 通常版は複数写真に対応: `["id_0.jpg", "id_1.jpg"]` 形式でJSON配列保存
- 簡易版は1枚: `id.jpg` 形式で単一ファイル名保存

- **リスト行:** 最大3枚の重なりサムネイル(28×28px) + 枚数バッジ

## 43. レシピ履歴採用時の明細重複防止（2026-05-18）

### 43-1. 背景

レシピ詳細の履歴プレビュー/採用操作後に、現在の `recipe_items` に履歴スナップショットの明細が追加され、明細が増殖するケースがあった。

`【OEM】ウニふりかけ` を確認したところ、現在明細は42件まで増えていた一方、保存済み履歴 v1〜v3 の `snapshot_items` は各20件で重複なしだった。
そのため履歴データ自体ではなく、履歴採用時の現在明細置換処理に問題があると判断。

### 43-2. 修正内容

- `app/api/recipe/save/route.ts`
  - `replaceAllItems` オプションを追加
  - 履歴採用や全新規保存時は `deletedItemIds` に依存せず、API側で `recipe_id` 単位の全削除を行ってから明細をinsert
  - 削除ID取得漏れやプレビュー状態のtemp IDにより、既存明細の上にスナップショット明細が追加される事故を防止
- `app/recipe/[id]/page.tsx`
  - 履歴採用時は `replaceAllItems: true` で保存APIを呼び出す
  - プレビュー中に上部の保存ボタンを押しても通常保存経路に入らず、履歴採用処理へ誘導
  - プレビュー中の上部ボタン表示を「戻る」「Ver.xを採用」に変更

### 43-3. 補足

直前に試した1g単価読み込み修正は、画面を開いただけで原価が再計算される危険があったため切り戻し済み。
今回の修正は履歴採用時の明細置換処理に限定する。

## 44. レシピ履歴UIの修正版ワークフロー化（2026-05-18）

### 44-1. 背景

従来の「+ 履歴メモ」方式では、正式版をベースに修正版レシピを作る際に、
「履歴メモを先に作るのか、修正後に作るのか」が分かりにくかった。
履歴作成・メモ入力・保存・復元が同じ場所に混ざり、バージョン管理として誤操作しやすいUIになっていた。

### 44-2. 新しい操作

  - 「修正版を作る」
  - レシピを編集
  - 必要なら変更メモを入力
  - 「正式版として保存」
- 過去版確認:
  - 「過去版」を開く
  - v番号をクリックしてプレビュー
  - 採用する場合だけ「Ver.xを採用して保存」

### 44-3. 修正内容

- `app/recipe/[id]/page.tsx`
  - 「+ 履歴メモ」ボタンと未確定履歴バッジを廃止
  - 「現在の正式版 vN」「修正版を作る」「変更メモ」「過去版 N件」にUIを分離
  - 修正版編集中は下部保存バーを緑系に変更し、「正式版として保存」と表示
  - 修正版保存後に `/api/recipe/versions` を呼び出し、保存後の状態を新しい正式版履歴として記録
  - 未保存変更がある状態で過去版をプレビューする場合は、破棄確認を表示
