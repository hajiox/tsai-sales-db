## 2. レシピ印刷レイアウト修正 (2026-02-15 〜 02-16)

### 2-1. 目的
レシピ詳細ページ (`/recipe/[id]`) のA4印刷レイアウトを改善する。

### 2-2. 実施した変更

####  A4横向き (Landscape) 対応
- **ファイル:** `app/recipe/[id]/page.tsx`
- **変更:** `@page { size: A4 portrait; }`  `@page { size: A4 landscape; }`
- **理由:** 5列（#, 名称, 基本(1), A, B）を収めるために横幅が必要

####  バッチ量列 A/B の表示修正
- **問題:** `400 (A)` と `800 (B)` の列がHTMLに存在するが印刷時に消える
- **原因:** `globals.css` の `@media print` で `.bg-gray-50` が `display: none !important` に設定されていた
- **対処:** 印刷用テーブルのth/tdから `bg-gray-50` クラスを全て削除
- **ヘッダー形式:** `{batchSize1} (A)`  `A ({batchSize1})` に変更

####  32行対応の余白圧縮
- **目標:** A4横1ページに最大32行の材料を収める
- **圧縮前:** ヘッダー161px + 行20px  最大22行
- **圧縮後:** ヘッダー56px + 行17px  最大39行（32行は余裕で収まる）
- **具体的な変更:**
  - タイトル: `text-2xl`  `text-sm`, `leading-none`
  - スペック情報（充填量表記量保存方法殺菌）を1行にまとめた
  - テーブルセルのpadding: `py-1`  `py-0`
  - フォントサイズ: `text-xs`  `text-[10px]`
  - グループ見出し製造メモの余白も全て圧縮
  - グローバルCSS印刷td paddingを上書き: `padding: 1px 4px !important`

####  材料経費グループの非表示
- `groupedItems.map`  `groupedItems.filter(g => g.type !== 'material' && g.type !== 'expense').map`
- 印刷時は原材料と経費グループを除外

## 8. Amazon手数料の諸経費からの分離 (2026-02-18)

### 8-1. 目的
Amazon手数料（販売価格の%）が諸経費に混在し、正確に計算されないレシピがあった。Amazon手数料を諸経費セクションから分離し、レシピごとにON/OFFで管理できるようにする。

### 8-2. 実施した変更

#### DBスキーマ変更
- **カラム追加**: `recipes`テーブルに`amazon_fee_enabled` (BOOLEAN, DEFAULT false) を追加
- **SQL**: `ALTER TABLE recipes ADD COLUMN IF NOT EXISTS amazon_fee_enabled BOOLEAN DEFAULT false;`

#### Amazon手数料トグルスイッチの追加
- Pricing Card（ダークテーマの価格表示カード）内にトグルスイッチを追加
- ON/OFF切替でAmazon手数料の有無をレシピごとに設定可能
- ON時: 販売価格（税込） x Amazon手数料率(%) = Amazon手数料額を自動計算
- 手数料率はデータベース管理画面のグローバル設定（localStorage）から取得

#### 原価計算ロジックの分離
- `getTotals()`関数を改修し、items内の旧「Amazon手数料」expenseを除外して計算
- `amazonFee`を独立した計算値として管理
- 原価合計 = 材料費 + Amazon手数料（フラグON時のみ）
- 粗利益の計算もAmazon手数料込みで正確に反映

#### 原価内訳UIの追加
- Amazon手数料ON時、原価内訳を表示:
  - 材料資材経費: XXX
  - Amazon手数料 (10%): XXX

## 9. psql (PostgreSQLクライアント) のインストール (2026-02-18)

### 9-1. 目的
Supabase REST API (PostgREST) はDDL（ALTER TABLE等のスキーマ変更）を実行できないため、DBマイグレーション時にブラウザでSupabaseダッシュボードのSQL Editorを開く必要があった。これは非効率でAIエージェントとの連携も困難だった。`psql`をローカルにインストールし、コマンドラインから直接SQLを実行可能にする。

### 9-2. インストール内容

| 項目 | 値 |
|------|------|
| ツール | psql (PostgreSQL 17.8) |
| インストール方法 | `winget install PostgreSQL.PostgreSQL.17` |
| インストール先 | `C:\Program Files\PostgreSQL\17\bin\` |
| PATH追加 | ユーザーPATHに追加済み |

### 9-3. 接続コマンド

```powershell
$env:PAGER=""; psql -h aws-0-ap-southeast-1.pooler.supabase.com -p 6543 -U postgres.zrerpexdsaxqztqqrwwv -d postgres --no-psqlrc -P pager=off -c "<SQL>"
```

### 9-4. 注意事項
- `PAGER`環境変数を空にする（Windowsに`cat`コマンドが無いためエラーになる）
- `--no-psqlrc`と`-P pager=off`を必ず付与
- UTF-8エンコーディング確保のため`[Console]::OutputEncoding`を設定
- ワークフロー: `.agent/workflows/db-migration.md`に手順を記載済み

### 9-5. 背景
Supabase REST APIが使えなかった理由:
1. PostgRESTはCRUD操作のみ対応。DDL（ALTER TABLE等）は実行不可
2. `rpc/exec_sql`のようなカスタム関数はデフォルトで存在しない
3. Supabase Management APIは別のアクセストークンが必要

## 10. AI原材料ラベル生成機能の実装 (2026-02-19)

### 10-1. 目的
食品表示法（2025年最新基準）に準拠した原材料名を自動生成し、商品ラベル作成業務を効率化する。また、複合原材料の成分データを一元管理し、表記の揺れや抜け漏れを防ぐ。

### 10-2. 実装内容

#### インライン生成・編集UI
- **ファイル:** `app/recipe/[id]/page.tsx`
- **変更:** 商品名の右側にインラインの原材料表示パネルを追加。
- **機能:**
  - **AI生成:** Gemini APIを使用し、レシピ構成から原材料名を自動生成。
  - **インライン編集:** 生成結果を確認・修正してから保存可能。自動保存は廃止し、安全性を重視。
  - **警告表示:** 不足データの警告（例: アレルギー情報の欠落）を表示。
  - **削除機能:** ゴミ箱アイコンで原材料表示をリセット。

#### Gemini AI プロンプト設計
- **ファイル:** `app/api/recipe/generate-label/route.ts`
- **ロジック:**
  - レシピ内の食材(ingredient)と中間加工品(intermediate)を展開し、DBの `raw_materials`（原材料構成）を取得。
  - 重量順、複合原材料の展開、添加物の分離（/以下）、アレルギー表示（末尾）などのルールを適用。
  - **安全性強化:**
    - 原材料データがない複合原材料（例: 〇〇麺、〇〇ソース）について、AIによる推測補完を**厳格に禁止**。
    - 代わりに `【要確認: 〇〇】` というプレースホルダーを出力し、ユーザーに手動確認を促す仕様に変更。これは食品事故防止のため必須の措置。

#### データベース連携
- **Fuzzy Matching (あいまい検索):**
  - レシピ上の食材名（例: "バーモンドカレー"）とDB上の正式名（例: "バーモントフレーク"）の表記ゆれを吸収。
  - 正規化（スペース削除、半角全角統一）に加え、ブランド名抽出やLCS（最長共通部分列）によるマッチングを実装。

## 18. レシピ管理機能の全面修正 — RLSバイパス・原価計算・テーブル間移動 (2026-02-26)

### 18-1. 目的

### 18-2. RLSバイパスのためのAPI移行

#### 新規API
| API | メソッド | 用途 |
|-----|---------|------|
| `/api/recipe/update` | PATCH | レシピフィールド更新（名前・カテゴリ・シリーズ・商品番号等） |
| `/api/recipe/update` | POST | レシピ複製（全アイテムのディープコピー） |

#### 移行対象ページ・操作
| ページ | 移行した操作 |
|--------|------------|

#### 設計方針
- `SUPABASE_SERVICE_ROLE_KEY` はバックエンドAPI内でのみ使用（フロントエンドに露出しない）
- RLSポリシーは読み取りには維持、書き込みのみAPI経由で管理操作

### 18-3. 資材・諸経費の原価計算修正

#### 修正したバグ
| # | 問題 | 原因 | 修正内容 |
|---|------|------|----------|
| 1 | 資材・諸経費の使用量入力欄がない | `isMaterialGroup` 条件で使用量欄を非表示（`-`）にしていた | 全タイプで使用量入力欄を表示、プルダウン選択時にデフォルト`usage_amount=1` |
| 2 | 資材の`unit_quantity`が文字列→コスト計算スキップ | materialsテーブルの`unit_quantity`がtext型（"3000"等） | マスター読み込み時に`parseFloat`で数値変換、デフォルト1 |
| 3 | 資材の原価が¥0になる（**根本原因**） | 食材と同じ`usage × (price / unit_quantity)`で計算 → ダンボール¥29.7÷3000=¥0 | **資材・経費**: `usage × price`（1個単価）、**食材**: `usage × (price / unit_quantity)`（g単価）に分岐 |
| 4 | 資材の400個分・800個分が`-`表示 | `isMaterialGroup`で非表示 | 「個」単位で表示に修正 |

#### コスト計算式の整理
```
食材:   cost = usage_amount(g) × (price(パック価格) / unit_quantity(パック入数g)) × tax_rate
資材:   cost = usage_amount(個) × price(1個単価) × tax_rate
諸経費: cost = usage_amount(回) × unit_price(1回単価) × tax_rate
```

### 18-4. 材料データベースのテーブル間移動機能

## 22. WEB/卸販売 単価スナップショット改修・BASE CSVインポート修正 (2026-03-04~05)

### 22-1. 目的
商品価格変更時に過去の売上データが最新価格で再計算される問題を解消し、BASEのCSVインポートでの金額不一致・マッチング問題を修正する。

### 22-2. WEB販売管理 — 単価スナップショット

#### 問題
商品マスターの価格を変更すると、過去月の売上金額も最新価格で再計算されてしまい、過去データの正確性が失われる。

#### 修正
- `web_sales_summary` に `unit_price` カラム追加（インポート時の単価を保存）
- `calculateTotalAllECSites()` のフォールバック: `unit_price → price → productMap.price`
- 過去データは過去の単価で、新規データは現在の単価で正確に計算

### 22-3. 卸販売管理 — 同等修正
- `wholesale_summary` にも `unit_price` カラムを追加
- WEB販売と同じスナップショット方式を適用

### 22-4. OEM売上入力画面 — マスターデータ優先方式
- **問題:** 単価入力フィールドがあるが、結局マスターデータで上書きされて分かりにくい
- **修正:** 単価フィールドをマスターから自動取得・読み取り専用に変更。「単価変更はOEM商品管理から」と注記
- OEMエリアサマリーは `sale.unit_price`（実売上データ）を表示

### 22-5. BASE CSVインポート修正

#### 合計金額の取得（base-parse ver.5）
- CSVの「合計金額」列（23列目）もパースし、商品ごとに金額を集計
- `base_amount` として `web_sales_summary` に保存（参考データ用）

#### BASEのみ複数マッチ許可（csvHelpers.ts）
- **問題:** BASEは「送料別」「送料込み」で同一商品を2つ販売するが、一方がマッチすると他方が宙ぶらりんになる
- **修正:** `channel === 'base'` の場合のみ `matchedIds` 重複排除をスキップ
- 他ECサイトは従来通り1:1マッチを維持

#### 金額表示の設計方針
- クーポン・割引は広告費として扱い、売上金額には影響させない
- 全ECサイト共通: `count × マスター価格 = 本来売り上がった金額` を表示
- `get_monthly_financial_summary` RPC関数も `count × products.price` に統一

## 36. 裏ラベルチェック: 簡易チェック・画像保存・閾値改修 (2026-03-22)

### 36-2. 簡易チェックモード

**概要:** DB照合をスキップし、OCR結果とユーザ入力の賞味期限で差異判定を行う簡略版。

- **モード選択画面:** 最初に「通常チェック」「簡易チェック」を選択

- **自由入力:** 数値のみ（例: `90`）は「日」として解釈するフォールバック追加
- **判定基準:** 25%差異（2年以上は35%）
- **ダッシュボード表示:** `⚡簡易` バッジで通常チェックと区別

### 36-3. 画像保存機能

**概要:** 通常版・簡易版の両方で撮影写真を自動保存し、ダッシュボードで閲覧可能にする。

#### 画像処理フロー
1. クライアント側: Canvas APIで最大1200pxにリサイズ → JPEG品質段階的低下 → **200KB以下**に圧縮
2. base64でAPIに送信（判定後バックグラウンドで実行、UIをブロックしない）
3. サーバー側: `data/label-images/{checkId}.jpg` に保存
4. DB: `label_checks.image_path` カラムに記録（単一ファイル名 or JSON配列）

#### 複数枚対応
- 通常版は複数写真に対応: `["id_0.jpg", "id_1.jpg"]` 形式でJSON配列保存
- 簡易版は1枚: `id.jpg` 形式で単一ファイル名保存

#### ダッシュボード表示
- **リスト行:** 最大3枚の重なりサムネイル(28×28px) + 枚数バッジ
- **展開詳細:** 「📷 撮影画像 (N枚)」として64×64pxの横並びギャラリー
- **クリック拡大:** 全画面モーダルで拡大表示、背景クリック or ✕ボタンで閉じる

### 36-4. 賞味期限判定閾値の改修

**変更:** 賞味期限が長い商品の許容差異を拡大。

| 賞味期限 | 日数 | 許容差異 |
|---|---|---|
| 60日〜18ヶ月 | 〜540 | 25% |
| **2年以上** | **730〜** | **35%** |

- **UI:** 結果画面の許容値表示を動的化（正規版・簡易版の両方）
- **理由:** 長期保存品は製造ロット差・在庫回転による日付のバラツキが大きいため

### 38-2. 原材料表示生成 v7 — 重複排除 + キャリーオーバー対応

#### 問題点（v6まで）
1. **原材料の重複**: 同じ食材（例: 食塩）がトップレベルで重複して表示される
2. **キャリーオーバー未考慮**: 全食材の全添加物を無条件に収集し、最終製品で効果のない微量添加物まで列挙

#### v7の改善内容

**原材料重複排除**:
- トップレベル原材料を`Set`で重複排除（カッコ内は個別に許容）
- 中間部品のサブ原材料も`Set`で重複排除

**キャリーオーバー自動省略**:
- 全体重量を計算し、各食材の重量比率を算出
- 食材が全体の5%未満 → その添加物は「キャリーオーバー候補」
- 同じ添加物が5%以上の主要食材からも検出される場合は表示を維持
- 全ての出現元が5%未満の場合のみ省略

**UI通知**:
- 省略された添加物を青色の「🔄 キャリーオーバー省略」パネルで表示
- 何が省略されたか確認可能

#### パイプライン（v7）
`食材分類 → raw_materialsパース → 添加物統合 → 重複排除 → キャリーオーバー除外 → ラベル組立 → AIレビュー`
